# Tech Stack Architecture (Global)
#project-concept

## Scope
This architecture covers the current client-only Gear Calc application: UI composition, calculator domain logic, rendering behavior, styling system, quality controls, and deployment workflow.

## Recommended Tech Stack
- **Runtime/UI**: Svelte 5 + Vite 7
- **Language**: TypeScript for domain modules, Svelte script blocks for UI orchestration
- **Testing**: Vitest baseline (expand with component/integration coverage)
- **Type/Static Checks**: `svelte-check`, TypeScript config validation
- **Styling**: CSS modules within Svelte components + global app stylesheet tokens

## Application Structure
- Single-page entry point (`src/App.svelte`) orchestrates app state and composition.
- Domain logic centralized under `src/lib/gear/`:
  - per-gear calculators (`calc/*.ts`)
  - shared types/config/editable/units/format/visualization utilities
- UI primitives and composed sections under `src/lib/ui/` grouped by purpose:
  - `layout/`, `gear/`, `inputs/`, `panels/`, `results/`, `visualizer/`
- Root-level docs anchor architecture and conceptual planning under `/neuroplast/`.

## Rendering Strategy
- Entirely client-rendered SPA behavior (no server rendering layer).
- Reactive recalculation loop in App-level state for deterministic updates.
- SVG visualizer renders from current normalized state per interaction cycle.

## Component Architecture
- **Container shell**: `App.svelte` manages selected gear, units, locks, warnings, results.
- **Control components**: type selector, unit toggle, number/lockable/module inputs.
- **Display components**: editable results panel, generic result cards, visualizer.
- **Separation rule**: calculators remain pure and UI-independent.

## State Management
- Local reactive state in `App.svelte` is source of truth.
- No global store required at current complexity.
- Derived reactive values govern editability, labels, and conversion modes.
- Lock maps represent user-intent constraints across input/result edits.

## Forms and Validation
- Number inputs with constraints (min/max/step) handled at component level.
- Domain validation provided by per-gear `validate()` functions returning warnings.
- Reverse-edit operations mediated by `applyResultEditToInputs` with guard rails.
- Future recommendation: standardized field-level validation messaging contract.

## Navigation and Routing Behavior
- No route segmentation; workflow remains in one page.
- Navigation is conceptual (section-to-section progression), not URL-based.
- Future deep-linking can be added via query serialization of input state.

## Design System
- Move to a dark-first tokenized palette in `:root` (surface, border, text, accent, warning).
- Apply consistent radius, spacing, and elevation scales across cards/panels.
- Use monospaced/tabular numerics for measured values and key readouts.
- Maintain visual hierarchy via restrained accent usage and semantic color roles.

## Accessibility Requirements
- Meet WCAG 2.2 AA contrast for text, controls, and status messaging.
- Ensure keyboard focus visibility on all interactive elements.
- Preserve semantic labels for inputs and role/aria labels for visualization.
- Avoid color-only encoding for warnings and active states.

## Security Considerations
- Client-only app with no auth/session secrets in current scope.
- Keep dependencies current and avoid unsafe dynamic HTML rendering.
- Maintain strict type discipline in domain calculations to reduce logic errors.
- Treat import/export/persistence additions as future security review triggers.

## Analytics and Observability
- Current scope has no telemetry.
- Future optional instrumentation points:
  - gear type change
  - unit mode change
  - result edit attempts/success
  - warning frequency by gear type
- Error surface strategy: capture runtime exceptions around calculation/visualization blocks.

## Performance Targets
- Initial interactive load target: <1.5s on modern desktop.
- Calculation response target: perceptually instant for standard input edits (<16ms typical).
- Visualizer render should remain smooth at common tooth-count ranges.
- Keep CSS and component logic lightweight; avoid heavy runtime dependencies.

## Testing Strategy
- Unit tests for each calculator module and shared conversion helpers.
- Integration tests for reverse-edit and lock interaction logic.
- Component behavior tests for critical controls (unit toggle, selector, editable results).
- Add accessibility checks for focus order, keyboard usage, and contrast validation.

## Environment Configuration
- Node.js 18+ runtime.
- Scripts:
  - `npm run dev`
  - `npm run check`
  - `npm run test`
  - `npm run build`
  - `npm run preview`
- No required secret environment variables for current architecture.

## Deployment Model
- Static frontend bundle generated by Vite to `dist/`.
- Deployable on static hosting/CDN platforms.
- No server runtime, database, or background jobs required.

## Architecture Decisions
- **Single-page workbench first**: minimizes cognitive/context switching for technical tasks.
- **Pure calculator modules**: enables deterministic, testable geometry outputs.
- **Client-only execution**: keeps latency low and deployment simple.
- **Dark technical aesthetic direction**: supports prolonged professional usage and product differentiation.
